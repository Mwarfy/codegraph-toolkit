## P0 — Decouverte majeure : default detector set tres limite

`codegraph analyze` sans `codegraph.config.json` n'active que **6
detecteurs** sur les 60+ disponibles dans le toolkit :

```
Detectors:  ts-imports, event-bus, http-routes, bullmq-queues, db-tables, block-loader
```

Tous les detecteurs avances (composite, security, taint, complexity,
dead-code, signature-duplication, magic-numbers, deprecated-usage,
sql-schema, drizzle-schema, eval-calls, hardcoded-secrets, ~~50 autres~~)
sont **opt-in via config**. Un user typique qui clone et lance ne voit
que ~10% du potentiel du toolkit.

**Pourquoi c'est genant** :

- L'experience "out-of-the-box" sous-vend massivement ce que le toolkit
  peut faire.
- Le user n'a aucun signal lui disant "tu pourrais activer X detecteurs
  de plus" — il pense que le toolkit est limite a structure + edges.
- Les outputs `synopsis-level{1,2,3}.md` ne mentionnent ni la
  configuration courante ni les detecteurs disabled.
- Les `phase38` summaries (DSM, packageDeps, barrels) sont peuples
  partiellement parce que la majorite des facts requises pour les rules
  composite ne sont jamais emis.

**Pistes de fix** (par ordre de complexite) :

1. **Ajouter un set par defaut "all-static"** qui active tous les
   detecteurs purement static (~40 sans dependances externes). Garde
   opt-out via `--detectors` ou `config.detectors`.
2. **Logger un hint** : "30+ detectors are disabled. Run
   `codegraph init --detectors all` to enable them" en fin d'output.
3. **Commande `codegraph init`** (deja propose dans EXTERNAL-AUDIT et
   F-010) qui detecte la stack + propose une config adaptee, incluant
   un set de detecteurs explicit.

Impact attendu : 5x plus de signaux remontes a un user qui essaie le
toolkit pour la premiere fois.

## P1 — Conventions OSS manquantes (faux positifs orphan systematiques)

Les 4 monorepos audites ont entre 16 et 145 orphans residuels. La
quasi-totalite tombent dans des **conventions OSS universelles** que le
detecteur ne connait pas :

| Convention | Projets affectes | Exemple |
|-----------|-------------------|---------|
| `examples/` | mcp-sdk (32), trpc (67), tanstack-query (75) | `examples/express-server/src/client.ts` |
| `benchmarks/` | hono (10) | `benchmarks/deno/fast.ts` |
| `runtime-tests/` | hono (2) | `runtime-tests/bun/index.test.ts` |
| `perf-measures/` | hono (1) | `perf-measures/jsx/dist.bench.ts` |
| `www/` (Next.js docs site) | trpc (30) | `www/src/components/Card.tsx` |
| `__testfixtures__/` (jscodeshift codemods) | tanstack-query (34) | `query-codemods/src/v4/__testfixtures__/default-import.input.tsx` |
| `__mocks__/` (vitest/jest) | tanstack-query (vue-query, 2) | `vue-query/src/__mocks__/useBaseQuery.ts` |
| `*.test-d.ts` (vitest type tests) | tanstack-query (svelte-query, 7) | `svelte-query/tests/createMutation.test-d.ts` |
| `*.svelte.ts` (Svelte 5 reactive) | tanstack-query (svelte-query, 1) | `svelte-query-persist-client/src/utils.svelte.ts` |
| `test-setup.ts` (variant generique) | tanstack-query (multiple) | `angular-query-experimental/test-setup.ts` |

**Action proposee** : etendre `core/framework-conventions.ts` avec une
nouvelle convention `isOssConventionEntryPoint(filePath)` qui matche :

```ts
const OSS_CONVENTION_DIRS = [
  'examples',     // standalone tutorials/demos shipped avec la lib
  'benchmarks',   // perf comparisons, runners independants
  'samples',      // synonyme examples
  'demos',        // synonyme examples
  'playground',   // exploration scripts
  'runtime-tests', // hono pattern
  'perf-measures', // hono pattern
  'www',          // site docs Next.js des projets OSS
  'website',      // synonyme www
]

const TEST_FIXTURE_PATTERNS = [
  /\/__testfixtures__\//,
  /\/__mocks__\//,
  /\.test-d\.[mc]?[tj]sx?$/,
  /\.svelte\.[mc]?[tj]s$/,
  /(?:^|\/)test-setup\.[mc]?[tj]s$/,
]
```

Impact : sur tanstack-query, ~145 → ~10 orphans (95% reduction).

## P2 — Support workspaces / monorepos absent

Trois projets audites sont des monorepos (mcp-sdk, trpc, tanstack-query)
gerent par pnpm workspaces / Turborepo / Lerna. Le toolkit ne les
reconnait pas automatiquement, ce qui cree deux types de bruit :

### P2.1 — `packages/*/` non workspace-aware → orphans

Files dans `packages/<sub>/src/...` qui sont importes par un autre
package via `workspace:*` deps sont marques orphans car le toolkit ne
suit pas les imports cross-workspace.

Exemple tanstack-query : `packages/svelte-query/tests/...test-d.ts`
imports `@tanstack/svelte-query` (qui est `packages/svelte-query/src`),
mais l'import passe par le nom du package npm pas par chemin relatif.

**Action proposee** :
- Detecter `pnpm-workspace.yaml`, `package.json#workspaces`,
  `lerna.json#packages` au boot de l'analyzer.
- Construire un map `name → path` depuis chaque `packages/*/package.json`.
- Lors du resolve d'un import `@scope/x`, si `name` matche le map, suivre
  vers le `src/` du package au lieu de `node_modules`.

### P2.2 — `package-deps` flag les workspaces internes comme `unused`

Stats des `unusedDeps` sur les monorepos :

| Projet | unusedDeps reportees | Vraies positifs (estim.) |
|--------|---------------------:|------------------------:|
| trpc | 257 | < 30 |
| tanstack-query | 345 | < 50 |
| mcp-sdk | 111 | < 20 |
| hono | 16 | ~10 |

Sur trpc, des deps comme `@trpc/react-query` (workspace interne) ou
`typescript`/`@cloudflare/workers-types` (build-time / type-only) sont
massivement reportes comme unused.

**Action proposee** :
1. **Workspace-aware** : si une dep `@scope/x` matche le name d'un
   workspace local, l'exclure de la verification (par definition,
   l'usage est conditionnel via le runtime).
2. **Type-only deps** : si `typescript.compilerOptions.types: ["X"]`
   inclut le package, OU si le package matche `@types/*`, exclure.
3. **Build-time deps whitelist** : `typescript`, `eslint`, `prettier`,
   `vitest`, `@vitejs/*`, `tsup`, `tsx`, `@types/*` etc. ne devraient
   pas trigger `unused-deps`.

Impact : reduction estimee de 60-80% des `unused-deps` sur monorepos.

## P3 — Detecteurs composite trop bruyants sur multi-framework libraries

Sur tanstack-query (1652 violations datalog total), repartition top-3 :

```
392 NO-FLOATING-PROMISE
387 COMPOSITE-ORPHAN-FILE
330 COMPOSITE-NEAR-DUPLICATE-FN
229 COMPOSITE-COPY-PASTE-FORK
```

`COMPOSITE-NEAR-DUPLICATE-FN` + `COMPOSITE-COPY-PASTE-FORK` cumulent
**559 violations**. Mais tanstack-query est intentionnellement une
multi-framework library qui ship des adapters identiques en
react-query / vue-query / svelte-query / solid-query / angular-query
— le code est volontairement parallele pour exposer la meme API.

**Action proposee** : ajouter une heuristique qui detecte quand deux
fichiers near-duplicate vivent dans des packages differents d'un meme
monorepo (e.g. `packages/react-query/src/foo.ts` ↔
`packages/vue-query/src/foo.ts`). Dans ce cas, c'est probablement un
adapter pattern intentionnel — flag avec un severity reduite ou
exempt par defaut, opt-in via config si le projet veut le check.

Aussi : `NO-FLOATING-PROMISE` 392 est tres haut. Probablement parce que
les `examples/` ont des `await fetch()` non gere — ce sont des demos
didactiques. Si on ajoute la convention `examples/` (P1), `NO-FLOATING-
PROMISE` chute fortement.

## P4 — Performance sur tres gros monorepos

`tanstack-query` (863 files) prend **43.4s** en analyse default. C'est
acceptable mais commence a etre long pour un workflow CI ou un agent
qui re-run apres chaque commit.

Repartition timing (estimee depuis happenin et logs disponibles) :
- `ts-imports` ~ 25-30% (parser TypeScript le plus lourd)
- `unused-exports` ~ 15-20%
- `complexity` / `function-complexity` ~ 10-15%
- `floating-promises` ~ 10-15% (multi-tsconfig)
- Le reste ~ 30%

**Pistes** (deja note dans EXTERNAL-AUDIT P2) :
- Mutualiser TypeChecker entre detecteurs ts-morph (deja planifie).
- Skip detecteurs hot via `--detectors=structure` pour mode rapide.
- Salsa caching pour le multi-run.

Pas urgent — bloquant uniquement si le toolkit vise des projets
massifs (>5000 files) ou un usage interactif (watch mode).

## P5 — Discoverability des detecteurs disponibles

Le user n'a aucun moyen de decouvrir les detecteurs disponibles depuis
le CLI. Les seules sources sont :

1. La doc README.md du toolkit
2. Lire `packages/codegraph/src/core/detectors/index.ts`

Aucune commande `codegraph detectors --list` ou `codegraph help
detectors`.

**Action proposee** : commande `codegraph detectors` (ou flag
`analyze --list-detectors`) qui imprime :
- Nom du detecteur
- Description courte (1 ligne)
- Status : enabled by default | opt-in
- Cout (light/medium/heavy)
- Facts produits

Permet aussi `codegraph init --interactive` qui propose de cocher.
